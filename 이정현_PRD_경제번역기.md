# PRD — 경제번역기 (Economy Translator)

> **문서 성격**: 제품 요구사항 정의서(Product Requirements Document). 이미 출시·운영 중인 서비스를 표준 PRD 포맷으로 정리한 것.
> **작성**: 이정현 · 2026-07-09
> **실서비스**: https://economy-translator.vercel.app
> **한 줄**: 매일 아침 한국 경제 뉴스를 경제 입문자의 언어로 번역해 자동 발행하는 웹 서비스.

---

## 1. 한 줄 요약

경제 뉴스는 매일 쏟아지지만, 입문자에게는 용어부터가 벽이다. 경제번역기는 매일 아침 그날의 한국 경제 뉴스를 수집해 "무슨 일인지 → 왜 중요한지 → 내 지갑에 무슨 영향인지"를 쉬운 말로 풀어 자동 발행한다. 사람이 매일 쓰지 않아도 되도록 수집–생성–품질 채점을 자동 파이프라인으로 운영한다.

---

## 2. 문제 정의 · 타겟 유저 · 페인포인트

### 타겟 유저
경제에 관심은 생겼지만 **어디서부터 시작할지 막막한 입문자.** (예: 사회초년생, 재테크를 막 시작한 사람, 뉴스를 봐도 무슨 뜻인지 몰라 넘겨온 사람.)

### 페인포인트
- 경제 뉴스는 **용어·맥락 전제가 많아** 입문자가 읽어도 남는 게 없다.
- 하루에 기사가 수백~수천 건 쏟아져 **뭐가 중요한지 스스로 못 고른다.**
- "쉽게 설명해준다"는 콘텐츠는 많지만 **매일·정시에 나오지 않거나**, 정확도가 들쭉날쭉하다.

### 이 제품이 미는 지점
"쉽게" + "매일 정시에" + "그날 진짜 중요한 것만 골라서" — 세 개를 동시에 만족시키는 것. 특히 **선정(그날 뭘 다룰지)** 과 **정시성**을 자동화의 핵심 과제로 본다.

---

## 3. 핵심 가설 · 가치 제안

### 가치 제안 (Value Proposition)
> **"어려운 경제를 쉬운 말로 번역한다."**

제품 이름 자체가 이 가치를 담는다. 기능 목록이 아니라 "왜 쓰는가"를 한 문장으로 못 박고, 모든 구성 결정을 여기에 맞춘다.

### 핵심 가설
1. 입문자는 **기사 원문이 아니라 "번역된 해설"** 을 원한다 → 개별 뉴스마다 5단 해설(무슨 기사 / 쉽게 / 왜 중요 / 한국 영향 / 핵심 포인트) 구조로 검증.
2. 입문자는 **선정을 대신 해주기를** 원한다 → 그날 가장 중요한 이슈를 골라주는 TOP3·헤드라인이 핵심 가치.
3. 진짜 가치가 있다면 **다시 온다** → 성공을 조회수가 아니라 재방문율로 측정(4장 참조).

---

## 4. 성공 지표

측정 원칙: **"서비스가 잘 도는가(건강 지표)"와 "서비스가 목적을 이루는가(북극성 지표)"를 분리해서 본다.** 건강 지표가 다 초록불이어도 북극성이 안 움직이면 "잘 돌지만 의미 없는" 상태로 판정한다.

### 북극성 지표 (North Star Metric)
- **재방문율(리텐션).** 신규 유입은 광고로 살 수 있지만 재방문은 못 산다 → 가장 정직한 가치 신호. 호기심 1회가 아니라 "진짜 도움이 됐나"를 대표.

### 건강 지표 (Health Metric)
- **발행 완료 시각** — 매일 아침 9시대에 브리핑이 완성되는가.
- **발행 실패율** — 자동 발행이 누락된 날이 있는가.
- **콘텐츠 품질 점수** — 채점기가 매기는 일일 점수(7장).

### SLA (스스로 건 운영 약속)
- **매일 아침 9시대 발행 완료.** 이 약속을 지키는지 매주 추적한다.
- ⚠️ 트레이드오프 기록: 초기 목표는 "9시 정각 전 완성"이었으나, 뉴스 재료가 오전 8→9시 사이에 쏟아진다는 것을 수집 로그로 확인(8시대 434건 vs 9시대 870건)하고, **완벽한 정시성보다 재료가 꽉 찬 품질을 택해** 생성 시점을 9시 이후로 옮겼다. "완벽함보다 매일 지켜지고 품질이 보장되는 기준"을 택한 결정.

## 5. 핵심 요구사항 (무엇을 · 왜)

### 5-1. 콘텐츠 구조 — 입문자용 번역 포맷
브리핑 1건은 다음으로 구성:
| 구성 | 역할 |
|---|---|
| 헤드라인 2줄 | 오늘 가장 중요한 한 가지 |
| 전체 요약 (3~5문단) | 오늘의 큰 그림 |
| 오늘의 한 줄 (공유 카드, 20~40자) | SNS 공유·핵심 압축 |
| 오늘의 용어 | 입문자 벽인 용어 1개 해설 |
| TOP3 뉴스 | 그날 가장 중요한 3건 |
| 분야별 건강진단 | 6개 분야별 핵심 뉴스 |
| 핵심 흐름 | 뉴스 간 연결 맥락 |
| 개별 뉴스 5단 해설 | 무슨 기사 / 쉽게 / 왜 중요 / 한국 영향 / 핵심 포인트 |

**왜 이 구조인가**: 입문자는 "요약"만으로는 맥락을 못 잡고, "원문"은 어렵다. 그 사이에서 *선정(TOP3) → 큰 그림(요약) → 낱개 해설(5단)* 의 계층으로 소화 난이도를 낮춘다.

### 5-2. 자동 발행 파이프라인
- **수집 → 생성 2단계 분리.** 한 함수에 다 넣으면 실행 시간 한도를 넘겨 죽는다(실측). 단계를 쪼개 매일 무인 발행되게 함.
- **이중화(폴백).** 주 생성(cron-job.org, 9:07)이 실패해도 보험 경로(Vercel, 9:20)가 대신 발행 → 발행 누락 자체를 구조적으로 차단.
- **멱등성(하루 1회 고정).** 두 경로가 같은 날 각각 생성해 내용이 바뀌거나 뉴스레터가 두 번 나가는 것을 방지 — "이미 오늘 발행됐으면 손대지 않는다".
- **타임아웃·재시도.** 모든 외부 호출(요약 AI·시세 조회)에 대기 한도와 1회 재시도 → "남의 서버는 늦을 수 있다"를 전제로 설계.

### 5-3. 선정 품질 — "그날 진짜 중요한 것"만
- **이슈 우선 선정.** 단순 지수 시황·환율 전망·신문 사설·해외 단독 뉴스는 제외하고, 기업·정책·생활물가·부동산·고용 등 "보통 사람이 체감할 이슈"를 고르게 함.
- **같은 사건 중복 제거(이중 방어).** 기계 규칙(제목 토큰 겹침 계수로 near-duplicate 탐지) + AI 규칙(프롬프트로 "TOP3는 서로 다른 사건") 두 겹.
- **코드 후처리 가드.** AI가 고른 TOP3를 코드가 한 번 더 검문해 시황·중복이면 다른 후보로 자동 교체 → 확률적 모델을 코드가 최종 보증.

## 6. 범위 밖 (스코프 컷) — 안 만들기로 한 것과 이유

> **안 만드는 결정도 기획이다.** 만들 수 있다고 다 만들면 빌드 트랩(검증 안 된 기능을 계속 찍어내는 함정)에 빠진다.

| 안 만든 것 | 이유 |
|---|---|
| **실패 알림 이메일(resend)** | 운영자가 메일함을 잘 안 본다 → 알림을 만들어도 못 본다. 대신 "매일 아침 사이트를 직접 확인"하는 일간 루틴으로 감지(고장 나면 날짜가 안 바뀜). 채널을 운영자 습관에 맞춘 결정. |
| **완전 자동 채점 조치 (미달 시 자동 재생성·무인 운영)** | 채점 AI도 틀릴 수 있어, 신뢰가 검증되기 전 자동 조치는 멀쩡한 결과물을 망칠 위험. "기록만 → 자동 재생성 → 무인 운영" 3단계로 나누고 각 단계 진입에 조건을 걸어 미룸. |
| **TOP3 long-tail 완벽 필터링** | temperature 무작위성 때문에 매번 다른 경계선 기사가 샘. 핵심 실패 부류(같은 사건·시황·칼럼)는 코드로 잡되, 나머지를 100% 쫓으면 무한 튜닝. 여기서 종료로 명시적 결정. |
| **카드뉴스 완전 자동 발행(7~10h 파이프라인)** | 인스타 업로드가 어차피 수동이라 완전 자동은 과투자 → 반자동(열기→저장→업로드)으로 결정. |

## 7. 리스크 · 대응

리스크를 **종류별로 나눠** 각각 다른 장치로 막는다. (시간 누락 ≠ 내용 오류 — 다른 문제이므로 다른 장치.)

| 리스크 | 대응 장치 |
|---|---|
| **발행 시간 누락** | 이중화(폴백) — 주 생성 실패 시 보험이 대신 발행. |
| **숫자 할루시네이션** (없는 수치 지어냄) | 프롬프트에 "원문에 없는 숫자 금지" 규칙 + 채점기 사실 항목 감시. |
| **식별자 왕복 결함** | AI에게 UUID를 되받아 쓰게 하지 않고 짧은 인덱스만 고르게 함 — 데이터 연결은 코드가 책임(확률적 결함이라 테스트로 안 잡히고 운영 중 드물게 터짐). |
| **선정 오류** (중요치 않은 걸 고름) | 이슈 우선 규칙 + 코드 후처리 가드. |
| **채점기 자체의 오판** | 채점기에 원문·후보풀(ground truth)을 반드시 첨부(안 주면 채점기도 지어냄) + 항목별 독립 채점(오판 전염 차단) + 최종적으로 사람 보정 단계(P3)를 로드맵에 둠. |

### 품질 채점기 상세 (LLM-as-a-Judge)
매일 자동 발행되는 콘텐츠라 사람이 다 검수할 수 없다 → 다른 AI가 매일 채점.
- **기준표 v2.1**: 총 10점 = 5항목(이해도·사실·선정·다양성·톤) × 0~2점. 재현성을 위해 0/1/2 앵커 고정, temperature 0.
- **채점 방법 분리**: *자동으로 잴 수 있는가* vs *올바른 걸 재는가*를 구분. 형식(글자 수·개수)은 **코드로 검사**(LLM은 글자 수 세기를 자주 틀림), 투자조언·지어냄은 **실격(안전선)**, 나머지 5항목만 LLM 채점.
- **ground truth 첨부 필수**: 사실·선정 채점에는 원문 기사·그날 후보 기사 풀을 반드시 함께 줌. 없으면 '판정 불가'로 처리.
- **경로 분리**: 채점은 발행과 완전히 분리(비동기) — 채점이 죽어도 발행은 안 막힘.
- **점진적 자동화 로드맵**: P0 기준표 → P1 자동 채점(완료) → P2 경보선 → P3 사람 보정 → P4 독자 신호(재방문)와 점수 대조.

## 8. 이후 학습 (출시 후 배운 것)

운영하며 실제로 겪고 반영한 것들. "배포=완료"가 아니라 며칠간 실제 출력을 모아 튜닝하는 것이 운영이라는 걸 확인한 사례.

- **첫 장애(6/12)**: 자동 발행 실패를 17분 만에 발견 → 원인이 외부 호출 지연(5분 초과)임을 당일 재현 → 근본 수정 배포까지 하루에 끝냄. 이후 모든 외부 호출에 타임아웃·재시도. 포스트모템으로 기록.
- **"두 번째 판이 더 낫다"(7/2)**: 사용자 관찰을 수집 로그(434 vs 870건)로 규명 → 겉 증상(덮어쓰기)이 아니라 "언제·몇 번 발행하나"를 재설계(멱등성).
- **채점기 오판(7/9)**: 채점기가 대조 자료 시점 어긋남으로 정상 브리핑을 0점·실격 오판 → 후보 풀을 발행 시점 기준으로 고정, 항목별 독립 채점으로 수정. → **"채점 AI도 틀린다"는 것 자체가 P3(사람 보정)의 존재 이유**임을 실증.
- **핵심 교훈**: 돈 되는 능력은 "최신 모델을 아는 것"이 아니라 "모델이 실무에서 사고 안 치도록 검증·통제 구조를 짜는 것". 경제번역기의 모든 결정이 이 한 줄로 수렴한다.

---

### 부록 — 이 PRD가 증명하는 것 (포트폴리오 관점)
0→1로 만든 게 아니라 **매일 굴러가는 AI 서비스를 "운영 기획"의 언어로 설계·측정·개선한** 사례. 타겟·가치 정의(기획)부터 파이프라인 안정성(운영), 품질 채점 체계(AI 품질 관리), 스코프 컷(우선순위 판단)까지 한 서비스 안에 담겨 있다.
